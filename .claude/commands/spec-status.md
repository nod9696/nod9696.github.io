ゲーム仕様書の整合性チェックと更新状況を確認してください。

## 概要
仕様書プロジェクトの現在の状態を分析し、セクション間の整合性チェック・ソース資料との鮮度比較・問題点の検出を行う。

## 引数
- `$ARGUMENTS` にプロジェクト名が含まれていればそれを使用
- 空の場合は `f:/Claude/specs/` 配下のプロジェクト一覧を表示して選択を促す

## 手順

### 1. マニフェスト読み込み
`f:/Claude/specs/{project}/manifest.json` を読み込む。
存在しない場合は「先に `/spec-init {project}` を実行してください」と案内。

### 2. ファイル鮮度チェック
manifest.json の `sources` に記載された各ソースファイルについて：
- `ls -l --time-style=long-iso {path}` でファイルの最終更新日時を取得
- manifest の各セクションの `updated` と比較
- ソースが仕様書より新しい場合「更新推奨」としてマーク

結果をテーブル形式で表示：
```
## 📊 ソース資料 鮮度チェック

| ソース | 種別 | ソース更新日 | 仕様書更新日 | 状態 |
|--------|------|-------------|-------------|------|
| character_sheet_polished_v2.md | character | 2026-03-27 | 2026-03-25 | ⚠️ 更新推奨 |
| novel_common_v2.txt | novel | 2026-03-23 | 2026-03-25 | ✅ 最新 |
```

### 3. 新規ファイル検出
manifest に未登録のファイルがプロジェクトディレクトリに存在するか検索：
- Glob でプロジェクトディレクトリ内の *.md, *.txt ファイルを検索
- manifest の sources に含まれないファイルを「新規検出」として報告

### 4. セクション間整合性チェック
全セクションファイルを Read ツールで読み込み、Agent ツールで整合性チェックエージェントを起動：

```
あなたはゲーム仕様書の品質管理担当です。
以下の仕様書セクションを横断的に分析し、整合性の問題を検出してください。

【セクション一覧】
{全セクションの内容}

【チェック項目】
1. **変数名の統一性**: フラグ・変数名が全セクションで一貫しているか
2. **キャラクター名の統一**: 名前・表記ゆれがないか
3. **シーン参照の整合**: シナリオで参照されるBG/CG/BGM/SEが、ビジュアル/サウンド仕様に定義されているか
4. **フラグの使用・定義一致**: シナリオで使用されるフラグがフラグ仕様に定義されているか、定義されたフラグが実際に使用されているか
5. **技術的整合**: システム仕様の技術選定と、各セクションが想定する機能に矛盾がないか
6. **欠落の検出**: ソース資料に記述があるが仕様書に反映されていない情報

【出力形式】
問題を重大度で分類して報告：
- 🔴 修正必須（矛盾・不整合）
- 🟡 要注意（表記ゆれ・軽微な不一致）
- 🟢 提案（改善可能な点）
- 📊 カバレッジ（ソース資料の反映率の概算）

結果をMarkdownで出力してください。
```

### 5. レポート出力
以下の形式で統合レポートを表示：

```markdown
# 📋 仕様書ステータスレポート: {project}
**チェック日時**: YYYY-MM-DD
**仕様書バージョン**: manifest.updated より

## セクション一覧
| セクション | バージョン | 更新日 | 状態 |
|-----------|----------|--------|------|
| 基本システム設計 | v3 | 2026-03-25 | ✅ |
| シナリオ・分岐設計 | v2 | 2026-03-22 | ⚠️ 更新推奨 |

## ソース鮮度
{ステップ2の結果}

## 新規ファイル検出
{ステップ3の結果}

## 整合性チェック結果
{ステップ4の結果}

## 推奨アクション
1. `/spec-update {project} characters` — キャラクターシート更新を反映
2. `/spec-update {project} flags` — 新規フラグの定義追加
```

### 6. 完了
レポートを表示して終了。ファイルへの保存は行わない（読み取り専用の診断コマンド）。
